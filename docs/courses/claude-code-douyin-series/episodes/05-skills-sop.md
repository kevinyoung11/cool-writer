# 05 我复制同一段提示词复制到像个快捷键

## 成片标题

同一段提示词复制 27 次后，我把它做成了 Skill

## 封面文案

复制三次，就该转流程了

## 成片脚本

我有一段提示词，复制了很多次。

一开始我还挺得意。

觉得这是我的高级经验。

后来复制到第 27 次，我感觉不对了。

我不是在用 AI。

我是一个活的快捷键。

每次 review，我都要粘这一大段：

不要只看代码风格。

先看功能有没有坏。

再看数据流有没有断。

再看有没有安全风险。

再看有没有漏测试。

不要夸我。

不要说“整体不错”。

发现问题要给证据。

没发现问题也要说剩下什么风险。

它每次都说：

“明白。”

下次我还得再说。

这件事很像带新人。

你每天早上都跟他说：

“咖啡机不要加洗洁精。”

他说记住了。

第二天你闻到一股泡沫味。

这时候你就知道，不能再靠口头提醒了。

得贴张纸。

在 Claude Code 里，这张纸就是 Skill。

Skill 不是很玄的东西。

它就是把一类经常做的事，写成固定流程。

以后再遇到这类事，不用每次重新交代。

提示词像临时微信。

Skill 像公司流程。

临时微信容易漏。

流程写得烂，也会稳定地坑你。

所以 Skill 关键不是“写长一点”。

是写清楚怎么做。

坏 Skill 长这样：

“请严格审查代码，保证质量。”

这句话不能说错。

但没法干活。

它就像你跟朋友说：

“帮我买点好吃的。”

朋友站在便利店门口，开始思考人生。

好 Skill 要具体。

比如代码审查：

先确认改动范围。

再看功能回归。

再看数据流。

再看安全和权限。

再看测试。

最后输出问题、证据、影响、怎么验证。

排障 Skill 也一样。

不要写“帮我看看哪里坏了”。

写：

先复现。

看日志。

查配置。

列假设。

一个个验证。

不要看到第一个报错就当根因。

这个很重要。

我见过太多这种情况：

日志写 connection refused。

AI 说数据库挂了。

最后一看，是环境变量没加载。

这就像人咳嗽，它建议你换肺。

Skill 还能拆成多个文件。

主文件写什么时候用。

检查清单放一份。

输出模板放一份。

团队标准再放一份。

这样比一段超长提示词好。

超长提示词到后面很容易互相打架。

前面写“严格审查”。

中间写“语气友好”。

后面写“不要太长”。

最后 AI 给你一句：

“整体不错，但建议重写。”

你说它错了吗？

它每句都听了。

就是组合起来像甲方反馈。

我现在判断要不要做 Skill，很简单。

同一段话复制三次。

同一个坑纠正三次。

同一类任务团队里每个人做法都不一样。

这三种情况，别再靠手感。

写成 Skill。

尤其团队里很有用。

否则每个人都有自己的神秘提示词。

一个人说“像资深工程师一样 review”。

一个人说“严格一点”。

一个人说“不要给面子”。

AI 看完都不知道自己是工程师、面试官，还是来吵架的。

不过 Skill 也别乱放东西。

项目长期规则放 `CLAUDE.md`。

独立调研交给 Subagent。

自动跑测试用 Hooks。

查数据库、GitHub、浏览器这类外部证据，用 MCP。

Skill 就管一件事：

这类活，以后按什么步骤干。

别写一个 Skill 叫“帮我变成全栈高手”。

那不是 Skill。

那是许愿。

## 拍摄梗点

- 人变成活的快捷键。
- 咖啡机加洗洁精，需要贴纸。
- “帮我买点好吃的”让朋友在便利店思考人生。
- connection refused 被诊断成换肺。
- “整体不错，但建议重写”像甲方反馈。
- Skill 叫“全栈高手”是许愿。

## 知识点藏法

- Skill：重复任务的固定流程。
- 好 Skill：触发场景、步骤、输出、禁区。
- 排障 Skill：复现、日志、配置、假设、验证。
- 多文件 Skill：主文件、清单、模板、标准。
- Skill 和 `CLAUDE.md`、Subagent、Hooks、MCP 的分工。

## 屏幕动作

- 展示一段反复复制的 review 提示词。
- 改成 Skill 文件。
- 展示多文件 Skill 目录。
- 展示团队把 Skill 放进版本控制。

## 可抄 Skill 草案

```text
name: strict-code-review
description: 审查本次代码改动里的真实风险

使用场景：
用户要求 review、检查 PR、审查 diff、找回归风险时使用。

流程：
1. 先确认本次改动范围；
2. 优先看功能回归、数据流断点、安全风险、缺失测试；
3. 再看可维护性和边界条件；
4. 每个问题给证据、影响和验证方式；
5. 没发现阻塞问题时，也说明剩余风险。

不要：
- 不要泛泛夸奖；
- 不要只提风格问题；
- 不要没证据就猜；
- 不要自动改代码。
```

## 结尾衔接

最后一集，我们把这些东西接起来。因为会用 Claude Code 还不够，得让它别每次都靠你提醒。
