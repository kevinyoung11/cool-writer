# 03 它昨天刚会，今天又不会了

## 成片标题

Claude Code 昨天刚会启动项目，今天又问我一遍

## 封面文案

AI 失忆不可怕，记错更麻烦

## 成片脚本

有一次我用 Claude Code 改项目，前一天都好好的。

它知道怎么启动。

知道怎么跑测试。

还知道有个目录不要乱碰。

我当时挺欣慰。

感觉这个同事终于带出来了。

第二天我打开同一个项目。

它问我：

“这个项目怎么启动？”

我看着屏幕，愣了两秒。

这感觉很像你昨天刚教会一个人用打印机。

今天他拿着纸问你：

“这台白色柜子是干什么的？”

Claude Code 不是故意装傻。

它有上下文，但上下文不是无限的。

你跟它聊天、让它读文件、跑命令、看日志，这些东西都会占地方。

你可以把上下文想成一张桌子。

刚开始桌面很干净。

README 放左边。

测试命令放右边。

项目规则贴在屏幕上。

聊着聊着，桌上开始出现报错日志、临时方案、旧需求、测试输出、你随口说的一句“这个之后再看”。

最后桌子满了。

它开始整理。

这就叫压缩。

压缩不是坏事。

问题是，整理的人是它。

你以为重点是“不要碰生产配置”。

它可能记成“讨论过配置”。

你以为“这个方案废弃了”很重要。

它可能记成“这个方案出现频率很高”。

所以 AI 最烦人的不是忘了。

忘了它会问。

最烦人的是记错了。

它会很自信。

还会引用历史资料。

你一看，资料确实是你给的。

只是它拿错年份了。

这时候 `CLAUDE.md` 就派上用场。

我现在会把它当成项目门口的小纸条。

不是写口号。

不写“请保持代码优雅”。

这种话很像贴在公司墙上的“追求卓越”。

看完大家都点头，干活没人知道怎么追。

我会写很具体的东西：

项目用什么栈。

怎么启动。

怎么测试。

缩进和导出习惯。

哪些目录是核心。

哪些文件不要提交。

哪些成功状态不能假装。

比如：

“不要提交 `.env.local`。”

“上传成功要有真实文件记录。”

“质量分不能硬编码。”

这种话土。

但有用。

`CLAUDE.md` 还有两层。

项目级的，放项目根目录，团队共享。

用户级的，放你自己的配置里，写个人偏好。

比如你喜欢中文回复，提交摘要要短一点，代码注释少一点。

别把团队规则塞到个人记忆里。

不然就变成：你电脑上 AI 很懂事，别人电脑上 AI 又重新做人。

除了 `CLAUDE.md`，还有一个习惯很重要。

每次资料多的时候，先让它分一下：

哪些是当前事实。

哪些是旧判断。

哪些只是待验证。

哪些已经过期。

这一步看起来慢，其实省时间。

因为复杂项目里，文档经常互相打架。

旧文档说“上传还没接存储”。

新文档说“存储接了，但转写没接”。

如果不先分清楚，它可能认真修一个已经修好的问题。

这种工作特别有画面感。

像你周末去公司加班，给已经离职的同事办入职。

最后，再说一个很接地气的判断。

如果一个会话已经聊到你自己都翻不到重点了，就别硬撑。

开新会话。

把真正有用的结论写进文档。

把长期规则放进 `CLAUDE.md`。

新会话里直接引用文件。

这不是清空记忆。

这是收拾桌子。

桌子上不是东西越多越专业。

有时候只是你该倒垃圾了。

## 拍摄梗点

- 昨天会启动，今天问“白色柜子是干什么的”。
- 上下文桌子堆满日志、旧需求、临时方案。
- “讨论过配置”误记成规则。
- 给已经离职的同事办入职。
- 开新会话 = 收拾桌子，不是失忆。

## 知识点藏法

- 上下文窗口：桌子能摊开的资料。
- 自动压缩：它自己整理桌子，可能整理错。
- `CLAUDE.md`：项目门口小纸条，不是企业文化墙。
- 项目级 vs 用户级记忆。
- 上下文冲突：旧事实、新事实、待验证要先分开。

## 屏幕动作

- 展示坏 `CLAUDE.md`：全是“保持优雅、注意质量”。
- 展示好 `CLAUDE.md`：命令、禁区、边界、验收。
- 展示让 Claude Code 分当前事实、旧判断、待验证。
- 展示引用文件而不是复制大段文字。

## 可抄提示词

```text
先别改代码，帮我整理上下文。
请把资料分成：
1. 当前事实；
2. 旧判断；
3. 待验证假设；
4. 已经过期、不要再采用的内容。

如果资料冲突，请说明冲突在哪里、你采用哪份、为什么。
最后列出适合写进 CLAUDE.md 的长期规则。
```

## 结尾衔接

下一集我想了个办法：一个 AI 会乱，那我多派几个 AI。后来发现，开会这件事，人类乱，AI 也乱。
