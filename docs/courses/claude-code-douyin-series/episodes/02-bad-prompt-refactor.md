# 02 我说加个暗黑模式，它差点把项目送走

## 成片标题

我说加个暗黑模式，Claude Code 差点把项目送走

## 封面文案

需求太短，AI 会自己脑补

## 成片脚本

我第二次用 Claude Code，已经谨慎一点了。

第一次我说“随便看看”，它差点开始上班。

这次我不说随便了。

我说得很具体：

“帮我加个暗黑模式。”

听起来没毛病。

谁没加过暗黑模式？

页眉放个按钮，点一下变黑，再点一下变回来。

我脑子里是这个。

Claude Code 脑子里不是。

它先找主题文件。

可以。

再找全局 CSS。

可以。

再看组件库。

也可以。

然后它开始创建 ThemeProvider、封装 hook、加 localStorage、改按钮、改空状态、改登录页、改后台页。

最后页面一打开，我沉默了。

它确实暗了。

暗得很彻底。

像项目停电了。

我问它：“为什么改这么多？”

它说：“为了保证视觉一致。”

这句话一出来，我就知道麻烦了。

很多开发事故，开头都很朴素：

“顺手处理一下。”

“为了保持一致。”

“这里我顺便优化了。”

“我发现可以重构。”

这几句话听起来像好事。

但你只是想换个灯泡，它开始重新设计小区供电。

Claude Code 不是没听懂。

它是听到的信息太少，自己补了太多。

“加个暗黑模式”在我嘴里是一件小事。

在它那里可能变成：

主题系统要不要统一？

组件状态要不要抽象？

用户偏好要不要持久化？

服务端渲染要不要避免闪屏？

无障碍对比度要不要一起处理？

这些问题都不是错的。

但不是每次都要一次全做。

我后来就学乖了。

复杂一点的事，不让它直接写。

先让它写计划。

Plan Mode 就是干这个的。

你可以把它理解成：先让它拿铅笔画方案，别直接上电钻。

它只能读项目、看文件、想方案，不先动代码。

一个靠谱计划，至少要讲清四件事。

它看了哪些文件。

它准备改哪些文件。

它不准备碰哪些地方。

它怎么证明自己改对了。

最后一点最容易漏。

AI 说“完成了”，不等于真的完成。

有时候只是它不打字了。

比如上传功能，不能只看页面弹了“上传成功”。

文件有没有真的进存储？

数据库有没有记录？

刷新页面还在不在？

失败了有没有提示？

这几个没看，就别说完成。

暗黑模式也一样。

按钮能点，不代表做完。

刷新后主题还在吗？

系统主题会不会覆盖？

文字对比度看得清吗？

只影响目标页面，还是把整个后台一起染了？

这些才是验收。

所以提示词别只写“帮我加个功能”。

多写几句。

你不用写得很高级。

就像给同事派活一样说清楚：

这次只改哪块。

别动哪块。

先给方案。

做完跑什么。

怎么才算真的好了。

Claude Code 很强。

但它不认识你脑子里的“就简单弄一下”。

你不说，它就按自己的理解简单。

它的简单，可能是重构。

## 拍摄梗点

- 暗黑模式变“项目停电”。
- “为了保持一致”变事故前摇。
- 换灯泡变重新设计小区供电。
- Plan Mode = 拿铅笔画方案，别直接上电钻。
- AI 说完成 = 它不打字了。
- “它的简单，可能是重构。”

## 知识点藏法

- 第一条提示词：目标、范围、禁区、验收。
- Plan Mode：先读、先想、先列方案。
- Explore -> Plan -> Code -> Verify -> Commit：通过一次暗黑模式返工讲。
- 验证标准：不只看 UI 成功，还看状态、失败、刷新、测试。

## 屏幕动作

- 展示短提示词：“帮我加个暗黑模式”。
- 展示更好的提示词。
- 展示 Plan Mode 计划里涉及文件和验收点。
- 展示运行测试或 lint。

## 可抄提示词

```text
先不要改代码。请进入 Plan Mode。

目标：
【我要做的功能】

范围：
只看和修改【相关目录/页面/接口】。

别动：
【不相关页面、生产配置、密钥、旧功能】

验收：
请说明成功路径、失败路径、刷新后状态，以及需要跑哪些测试或检查。

先给计划，我确认后再实现。
```
