# 06 别找神提示词了，先把流程搭起来

## 成片标题

Claude Code 真正好用，是从“写代码”变成“跑流程”

## 封面文案

AI 写得快，不代表项目没坑

## 成片脚本

我用 Claude Code 用到后面，发现一件挺现实的事。

它最爽的能力，是写代码。

但最值钱的能力，不是写代码。

因为写代码太容易让人上头。

你说一句，它改一片。

再说一句，又改一片。

看着很像效率起飞。

过两天你打开项目，发现确实起飞了。

只是落点不太好。

只让 Claude Code 写代码，会留下几个麻烦。

第一个麻烦：它不知道外面的状态。

它看代码觉得上传成功。

但文件可能没进存储。

它看接口觉得任务创建了。

但后台任务可能没人消费。

它看页面写着 86 分。

但这个 86 可能只是某个地方写死了。

第二个麻烦：它改完不一定真验证。

它说“完成了”。

有时候这句话的意思只是：

“我暂时没话说了。”

第三个麻烦：你每次都要提醒它。

记得跑测试。

记得 lint。

别改密钥。

别展示假成功。

别顺手重构。

提醒多了之后，你会觉得自己不是开发者。

你是项目里的报站员。

所以最后要搭的不是一条神提示词。

是流程。

这个流程不用说得很玄。

就六样东西。

`CLAUDE.md`，管长期记忆。

项目怎么跑、哪些文件别碰、什么叫成功，写里面。

Plan Mode，管动手前先想。

先让它看文件、列方案、说风险。

Subagents，管单独的小任务。

长文档、调研、审查，能丢出去。

Skills，管重复流程。

review、排障、发布检查，别每次复制粘贴。

MCP，管外部证据。

数据库、GitHub、浏览器、文档，不要全靠你截图搬运。

Hooks，管容易忘的动作。

保存后格式化，提交前 lint，核心逻辑改了跑测试。

这几个东西放一起，Claude Code 就不只是“会写代码”。

它开始像一个被流程管住的助手。

注意，是被管住。

不是放飞。

MCP 不是让它到处乱逛。

是让它在授权范围里查证据。

没有 MCP，你就会变成 AI 的搬运工。

一会儿贴数据库截图。

一会儿贴 PR 状态。

一会儿贴浏览器报错。

贴到最后，你都忘了自己本来要做什么。

Hooks 也不是越多越好。

我见过最烦的自动化，就是改一个文案，它开始全量构建。

保存一个 Markdown，它开始跑端到端测试。

你关电脑，它像临终托孤一样给群里发通知。

这不叫自动化。

这叫电子惊吓。

好的 Hooks 要分层。

小改动跑小检查。

核心代码跑相关测试。

提交前跑 lint 和单测。

发布前再跑完整构建。

失败的时候给摘要，别把一整屏日志扔过来。

日志不是惩罚工具。

完整流程大概是这样：

你提任务。

它先读 `CLAUDE.md`。

进入 Plan Mode 看代码。

不确定的地方让 Subagent 去调研。

重复检查用 Skill。

它小步改代码。

Hooks 自动跑验证。

MCP 查外部状态。

最后给你一句人话：

改了什么。

怎么验证的。

还有哪里没把握。

这时候你会发现，Claude Code 没替你做老板。

它只是把很多杂活做掉了。

你还是要判断方向。

你还是要看风险。

你还是要决定要不要合。

但它可以少让你当搬运工、报站员、复制粘贴机。

所以别再问“有没有一句提示词，让 Claude Code 一次写对”。

更靠谱的问题是：

它写错的时候，你能不能发现？

它写对的时候，你能不能证明？

能做到这两点，Claude Code 才算真进了你的工作流。

## 拍摄梗点

- 效率起飞，但落点不太好。
- AI 说完成 = 暂时没话说。
- 用户变项目报站员。
- MCP 缺失，用户变 AI 搬运工。
- Hooks 乱配 = 电子惊吓。
- 日志不是惩罚工具。
- 少当搬运工、报站员、复制粘贴机。

## 知识点藏法

- `CLAUDE.md`：长期记忆。
- Plan Mode：动手前先看和计划。
- Subagents：隔离调研和审查。
- Skills：重复流程。
- MCP：查外部证据。
- Hooks：自动触发验证。
- 验证链：代码、运行结果、外部系统、用户可见状态。

## 屏幕动作

- 展示六块流程图：`CLAUDE.md / Plan Mode / Subagents / Skills / MCP / Hooks`。
- 展示一次 Hook 跑测试。
- 展示 MCP 查 PR 或数据库状态。
- 展示最后的验证摘要。

## 可抄总模板

```text
按这个流程处理任务：
1. 先读 CLAUDE.md 和相关文档；
2. 进入 Plan Mode，只读看代码，先别改；
3. 列出涉及文件、风险、别碰的地方和验收标准；
4. 不确定的问题用 Subagent 调研或审查；
5. 高频检查用对应 Skill；
6. 小步改代码；
7. 跑相关测试、lint、build 或 eval；
8. 如果有 MCP，核实外部系统状态；
9. 最后用人话说明：改了什么、怎么验证、还有什么风险。
```

## 系列收束

评论区回复“工作流”，我把这 6 集里的提示词、Skill、Subagent 和 Hooks 清单整理成一份能直接用的版本。
