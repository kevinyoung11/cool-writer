# 接地气改写标准

## 这版要避开的味道

- 别像老师讲课：少用命令式、条规式开头。
- 别像管理培训：少说“体系、闭环、长期能力、工程化”这类大词。
- 别像 AI 总结：少用抽象收束和上价值句。
- 别硬凹网感：不用“绝绝子、谁懂啊、狠狠”等空口语。

## 这版要靠近的感觉

像朋友坐在旁边吐槽：

> 我当时真以为它只是看看，结果它把 README、配置、测试全翻了一遍，还问我要不要运行命令。我手都停住了。

像真实操作里的小尴尬：

> 我说加暗黑模式，它给我弄出一个深蓝后台。我看了一眼，感觉不是夜间模式，是项目停电了。

像短视频里能被记住的句子：

> AI 说完成，有时候只是它不打字了。

## 改写动作

1. 把抽象句换成现场动作
   - 差：权限边界需要明确。
   - 好：别在 `Documents` 根目录开它。你一开，它可能看到五个项目和三个旧坑。

2. 把命令式换成踩坑后反应
   - 差：复杂任务不要直接开写。
   - 好：我后来学乖了，复杂任务先让它写计划，不然它手太快。

3. 把技术词翻成生活话，再保留术语
   - `上下文窗口`：它桌面上能同时摊开的资料。
   - `Plan Mode`：先让它拿铅笔画方案，别直接上电钻。
   - `MCP`：少给 AI 当搬运工，让它自己去授权工具里查证据。
   - `Hooks`：别靠记性跑测试，把容易忘的步骤绑到流程上。

4. 每个知识点都落到一个具体麻烦
   - 权限：它能不能碰文件。
   - 上下文：它会忘，也会记错。
   - Subagents：人多会乱，AI 多了也一样。
   - Skills：复制三次的提示词就该转成流程。

## 成片口吻

- 多用“我当时”“我后来”“你可能也遇到过”。
- 技术结论不要端着，尽量用一句普通话落地。
- 允许短句，但不要全是短句。
- 包袱要来自真实开发场景，不靠硬段子。

## 保留底线

- 知识点不能丢。
- 术语可以保留，但要解释成人话。
- 项目实践继续弱化，只作为“我打开了一个复杂项目”的背景。
- 每集结尾给可抄提示词，但正文别写成提示词课。
